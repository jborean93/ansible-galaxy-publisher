# Ansible Galaxy Publisher API - Implementation Design

_Note: This has been generated by an LLM iteratively, while it has been reviewed it may not be 100% accurate and/or out of date, use this as a guide only or reference for historical purposes._

## Executive Summary

This document describes the implementation design for a **transparent authenticated proxy** for Ansible Galaxy/Automation Hub APIs. The proxy:

1. Acts as a drop-in replacement Galaxy server endpoint for publishing a collection artifact
2. Authenticates GitHub Actions workflows using **GitHub OIDC tokens** (instead of Galaxy username/password/token)
3. Authorizes only **pre-configured repositories** to publish
4. Protects **sensitive Galaxy/Automation Hub tokens** (kept server-side, never exposed)
5. Transparently proxies all requests to the actual Galaxy/Automation Hub server
6. Works with existing tools (`ansible-galaxy` CLI, custom scripts) unchanged

**Key Benefit:** Users configure `ansible-galaxy` to point to our proxy URL and use their GitHub OIDC token for authentication. Everything else works exactly as before - no custom API client needed.

**Architecture:** Transparent HTTP proxy that intercepts Galaxy API requests, validates OIDC authentication, and forwards to real Galaxy servers with secure tokens.

## Core Design: Transparent Proxy

### How It Works

**User Perspective:**
```bash
# Instead of:
ansible-galaxy collection publish collection.tar.gz \
  --server https://galaxy.ansible.com \
  --token $GALAXY_SECRET_TOKEN

# Users do:
ansible-galaxy collection publish collection.tar.gz \
  --server https://our-proxy.example.com/api/v1/galaxy/ \
  --token $GITHUB_OIDC_TOKEN
```

**Proxy Flow:**

```
1. Client Request:
   POST https://our-proxy.example.com/api/v1/galaxy/api/v3/artifacts/collections/
   Authorization: Token eyJhbGc...  (OIDC token)
   Body: collection.tar.gz

2. Proxy Processing:
   - Extract server_id from URL path (`/api/v1/{server_id}/...`): "galaxy"
   - Extract remaining path: "/api/v3/artifacts/collections/"
   - Extract OIDC token from Authorization header
   - Validate OIDC token (signature, claims)
   - Extract repository from token claims
   - Check repository authorization
   - Look up server configuration for "galaxy" from pre-defined server
   - Get real Galaxy URL and token

3. Proxied Request:
   POST https://galaxy.ansible.com/api/v3/artifacts/collections/
   Authorization: Token <real-galaxy-token>
   Body: collection.tar.gz (unchanged)

4. Proxy Response:
   - Receive Galaxy's response
   - Return to client unchanged
   Status: 202 Accepted
   Body: {"task": "/api/v3/tasks/abc-123/"}
```

**Status Polling Works Too:**

```
Client: GET https://our-proxy.example.com/api/v1/galaxy/api/v3/tasks/abc-123/
Proxy:  GET https://galaxy.ansible.com/api/v3/tasks/abc-123/
        (with real token)
Response: Galaxy's task status (unchanged)
```

**Only Three Galaxy Endpoints Supported:**

The proxy handles only these Galaxy API endpoints:
- `GET /health` - Health check with service announcements (proxy-specific, no auth)
- `GET /api/` - API discovery (required by Galaxy CLI, no collection validation)
- `POST /api/v3/artifacts/collections/` - Upload collection (with collection validation)
- `GET /api/v3/imports/collections/{id}/` - Check import task status (no collection validation)
- All other endpoints return `404 Not Found`

**Note:** `/api/v3/tasks/{id}/` does NOT exist in Galaxy v3 API - only the import-specific status endpoint exists.

## Benefits of Transparent Proxy

**1. Works with Existing Tools**
- `ansible-galaxy` CLI works unchanged
- Custom scripts work unchanged
- Any HTTP client works
- No need to write custom client code

**2. Simpler Implementation**
- No custom request/response handling
- Just proxy logic: validate auth → forward request → return response
- Fewer edge cases to handle
- Less code to maintain (~100 lines)

**3. Focused Security**
- Only proxies publish and status check endpoints (minimal attack surface)
- Read operations (listing, browsing) not exposed through proxy
- Reduces complexity and potential vulnerabilities

**4. Better User Experience**
- Familiar `ansible-galaxy` commands
- Same workflow, just different URL and token
- Error messages come directly from Galaxy (more helpful)

## URL Structure

### Proxy URL Format

```
https://our-proxy.example.com/api/v1/{SERVER_ID}/{GALAXY_PATH}
```

**Components:**
- **Base:** `https://our-proxy.example.com/api/v1/`
- **Server ID:** Identifies which Galaxy server (e.g., `galaxy`, `automation-hub`)
- **Galaxy Path:** Remaining URI path (passed through to Galaxy)

**Examples:**

| Client Request | Server ID | Galaxy Path | Proxied To |
|----------------|-----------|-------------|------------|
| `/api/v1/galaxy/api/` | `galaxy` | `/api/` | `https://galaxy.ansible.com/api/` |
| `/api/v1/galaxy/api/v3/artifacts/collections/` | `galaxy` | `/api/v3/artifacts/collections/` | `https://galaxy.ansible.com/api/v3/artifacts/collections/` |
| `/api/v1/galaxy/api/v3/imports/collections/abc-123/` | `galaxy` | `/api/v3/imports/collections/abc-123/` | `https://galaxy.ansible.com/api/v3/imports/collections/abc-123/` |

### Path Extraction

**Algorithm:**
1. Match URL against pattern: `/api/v1/{server_id}/{galaxy_path}`
2. Extract `server_id` (first path component after `/api/v1/`)
3. Extract `galaxy_path` (everything after `/{server_id}`)
4. Look up server configuration for `server_id`
5. Construct target URL: `{server_base_url}{galaxy_path}`
6. Forward request with real Galaxy token

**Example:**
```
Request URL: /api/v1/galaxy/api/v3/artifacts/collections/
Pattern: /api/v1/{server_id}/{galaxy_path}
Extracted: server_id = "galaxy", galaxy_path = "/api/v3/artifacts/collections/"
Config lookup: "galaxy" -> base_url = "https://galaxy.ansible.com"
Target URL: https://galaxy.ansible.com/api/v3/artifacts/collections/
```

## Supported Endpoints

**This proxy supports three specific Galaxy API endpoints plus a health endpoint:**

### 1. Health Check (Proxy-Specific)
```
GET /health
```
- **Method:** GET only (case-sensitive)
- **Path:** Exact match
- **Authorization:** None required (public endpoint)
- **Purpose:** Health/readiness check for monitoring and displaying service announcements
- **Response Format:**
  ```json
  {
    "status": "healthy",
    "service": "galaxy-publisher",
    "version": "0.1.0",
    "messages": [
      {
        "level": "warning",
        "message": "Scheduled maintenance on 2026-04-01 00:00 UTC"
      },
      {
        "level": "info",
        "message": "New feature: Wildcard collection support available"
      },
      {
        "level": "deprecation",
        "message": "Support for Galaxy v2 API will be removed in v2.0.0"
      }
    ]
  }
  ```
- **Message Levels:**
  - `info` - General informational messages
  - `warning` - Important notices (upcoming changes, maintenance)
  - `deprecation` - Features being deprecated or removed
  - `error` - Service issues or degraded functionality
- **Note:** Clients can check this endpoint to display announcements. A GitHub Actions workflow can be created to check for new messages and create issues/notifications.

### 2. API Discovery
```
GET /api/
```
- **Method:** GET only (case-sensitive)
- **Path:** Exact match with trailing slash
- **Authorization:** JWT validation + authorization rules (NO collection validation)
- **Purpose:** Used by `ansible-galaxy` CLI to discover available API endpoints
- **Process:**
  1. Validate OIDC token
  2. Check authorization rules
  3. Proxy to Galaxy with server token
- **Response:** Galaxy's API version/endpoint information
- **Note:** Required for Galaxy CLI compatibility

### 3. Publish Collection
```
POST /api/v3/artifacts/collections/
```
- **Method:** POST only (case-sensitive)
- **Path:** Exact match with trailing slash
- **Authorization:** Full JWT validation + authorization rules + collection name validation
- **Request Body:** Collection tarball (`.tar.gz`)
- **Process:**
  1. Validate OIDC token
  2. Check authorization rules
  3. Extract `MANIFEST.json` from tarball
  4. Validate `namespace.name` against `allowed_collections`
  5. Proxy to Galaxy with server token
- **Response:** Galaxy's response (typically 202 Accepted with import task URL in `task` field)
  ```json
  {
    "task": "/api/v3/imports/collections/abc-123-def-456/"
  }
  ```

### 4. Import Task Status Check
```
GET /api/v3/imports/collections/{task_id}/
```
- **Method:** GET only (case-sensitive)
- **Path:** Pattern match `/api/v3/imports/collections/*/` (any UUID task ID)
- **Authorization:** JWT validation + authorization rules (NO collection validation)
- **Purpose:** Check the status of a collection import/publish operation
- **Process:**
  1. Validate OIDC token
  2. Check authorization rules
  3. Proxy to Galaxy with server token (no collection validation)
- **Response:** Galaxy's import task status response
  ```json
  {
    "id": "abc-123-def-456",
    "state": "completed",
    "finished_at": "2024-01-01T00:00:00.000000Z",
    "messages": [
      {"level": "INFO", "message": "Collection published successfully"}
    ]
  }
  ```
- **Note:**
  - Task URL is returned from the publish endpoint in the `task` field
  - Any authorized client can check any import status (no task tracking/ownership)
  - This is the ONLY status endpoint - `/api/v3/tasks/{id}/` does NOT exist in Galaxy v3 API

### All Other Endpoints
```
Any other path or method → 404 Not Found
```

**Examples of unsupported endpoints:**
- `GET /api/v3/collections/` → 404
- `GET /api/v3/tasks/{id}/` → 404 (this endpoint doesn't exist in Galaxy v3)
- `HEAD /api/v3/imports/collections/{id}/` → 404 (wrong method)
- `POST /api/v3/artifacts/collections` → 404 (missing trailing slash)
- Any other Galaxy API endpoint → 404

**Rationale:**
- **Security:** Minimize attack surface by only proxying what's needed
- **Simplicity:** No need to track task ownership or implement complex read operations
- **Purpose:** This proxy exists to secure publishing, not as a general Galaxy proxy

**Reference:** See [Galaxy API implementation](https://github.com/ansible/ansible/blob/devel/lib/ansible/galaxy/api.py) for official endpoint details

## Authentication

### OIDC Token as Galaxy Token

**Client sends:**
```
Authorization: Token eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6IjEyMyJ9...
```

**Note:** This matches Galaxy's token format (`Token <value>`), so `ansible-galaxy` CLI works unchanged.

### JWT Validation Flow

**1. Extract Token from Authorization Header**
- Parse `Authorization` header
- Expect format: `Token <token-value>`
- Extract token value

**2. Detect Token Type**
- If token is JWT format (3 base64 parts separated by `.`): OIDC token
- Otherwise: Reject (we only accept OIDC tokens, not real Galaxy tokens)

**3. Decode JWT (Without Verification Yet)**
- Split JWT into parts: `header.payload.signature`
- Base64-decode header and payload (signature verified later)
- Parse as JSON
- **Do NOT trust claims yet** - signature not verified

**4. Extract and Validate Issuer**
- Extract `iss` claim from JWT payload
- Look up issuer in configuration:
  ```
  For each configured issuer in oidc_issuers:
    If issuer.issuer_url == jwt['iss']:
      Use this issuer config
      Break
  ```
- If no matching issuer found: **Reject token** (unknown issuer)
- This prevents tokens from untrusted OIDC providers

**5. Fetch JWKS (Public Keys)**
- Get `jwks_url` from matched issuer config
- Example: `https://token.actions.githubusercontent.com/.well-known/jwks`
- JWKS contains public keys for signature verification

**6. Verify JWT Signature**
- Extract `kid` (key ID) from JWT header
- Find matching key in JWKS (where `jwks['keys'][].kid == jwt_header['kid']`)
- Verify signature using:
  - Algorithm from JWT header (e.g., RS256)
  - Public key from JWKS
  - Signature from JWT
- If signature invalid: **Reject token** (tampered or forged)
- **Now we can trust the claims**

**7. Validate Standard Claims**
- `iss` (issuer): Already validated in step 4
- `aud` (audience): Must equal `settings.oidc_audience` from config
- `exp` (expiration): Must be > current time (token not expired)
- `nbf` (not before, optional): If present, must be <= current time
- `iat` (issued at, optional): Sanity check (not too far in past/future)

**8. Extract Claims for Authorization**
- Extract claims needed for authorization (e.g., `repository`, `ref`, `workflow`)
- Store claims for next step

**9. Find Matching Authorization Rule**
- Iterate through `authorization` rules in config
- For each rule:
  - Check if `rule.oidc_issuer` matches the issuer identifier
  - Check if ALL claims in `rule.claims` match JWT claims (with wildcard support)
  - If match found: Use this rule for authorization
- If no match: **Reject** (no authorization rule for this token)

**10. Verify Server Access**
- Check if requested `server_id` exists in `rule.servers`
- If not: **Reject** (not authorized for this server)

**11. Success**
- Token validated and authorized
- Claims extracted
- Authorization rule identified
- Proceed to namespace validation and proxying

---

### Why Extract `iss` Before Verification?

**Security Note:** We extract `iss` from the JWT payload BEFORE signature verification, but we DON'T trust it yet. Here's why this is safe:

1. **Purpose**: We need `iss` to know which JWKS to use for verification
2. **Trust**: We only use it to *select* a JWKS endpoint, not to make authorization decisions
3. **Validation**: After signature verification, we validate `iss` matches the expected value
4. **Attack Prevention**:
   - Attacker can't forge a token from `https://evil.com` and claim `iss: https://token.actions.githubusercontent.com`
   - Signature verification would fail (signed with evil.com's key, not GitHub's)
   - Attacker can't use a real GitHub token for a different repository
   - Authorization rules check specific claims (repository, ref, etc.)

**Best Practice:**
1. Extract `iss` (untrusted)
2. Look up JWKS URL based on `iss`
3. Verify signature with JWKS
4. NOW trust `iss` and other claims
5. Match against authorization rules

## Configuration

### Configuration File: `config/servers.yml`

**Format:** YAML (YAML is a superset of JSON, so JSON also works if preferred)

**Benefits of YAML:**
- Supports comments for documentation
- More human-readable
- Can still use JSON syntax if desired

```yaml
version: 1

settings:
  # OIDC audience (must match token's aud claim)
  oidc_audience: https://our-proxy.example.com

# OIDC Issuer definitions (JWT validation configs)
oidc_issuers:
  github_actions:
    issuer_url: https://token.actions.githubusercontent.com
    jwks_url: https://token.actions.githubusercontent.com/.well-known/jwks

  # Not part of initial MVP but shows how it could be done in future
  gitlab_oidc:
    issuer_url: https://gitlab.com
    jwks_url: https://gitlab.com/oauth/discovery/keys

# Server definitions (endpoints and auth mechanisms)
servers:
  galaxy:
    base_url: https://galaxy.ansible.com
    token: GALAXY_TOKEN  # Environment variable name for the Galaxy API token

  automation-hub:
    base_url: https://console.redhat.com/api/automation-hub
    oauth_secret:
      auth_url: https://sso.redhat.com/auth/realms/redhat-external/protocol/openid-connect/token
      client_id: 03be8f22-6d6e-46f5-9740-3e3f7a16eced
      client_secret: AUTOMATION_HUB_SECRET  # Environment variable name for OAuth client secret

# Authorization rules
authorization:
  # GitHub Actions workflow for ansible-collections/community.name
  - oidc_issuer: github_actions  # References oidc_issuers.github_actions
    claims:
      repository: ansible-collections/community.name  # Exact match
      ref: refs/tags/v*  # Wildcard: any tag starting with v

    # Defines the servers and collections allowed on each server
    servers:
      galaxy:
        allowed_collections: ["community.name"]
      automation-hub:
        allowed_collections: ["ansible.name"]

  # GitHub Actions workflow for ansible-collections/ansible.windows (tags only)
  - oidc_issuer: github_actions
    claims:
      repository: ansible-collections/ansible.windows
      ref: refs/tags/v*  # Only tags starting with 'v'
      workflow: .github/workflows/publish.yml  # Specific workflow

    servers:
      galaxy:
        allowed_collections: ["community.general"]

  # GitLab CI example
  - oidc_issuer: gitlab_oidc
    claims:
      project_path: myorg/ansible-collections/*  # Wildcard: any project under myorg
      ref: main

    servers:
      galaxy:
        allowed_collections: ["myorg.name"]
```

### OIDC Issuer Configuration

**Purpose:** Define OIDC providers and their JWT validation endpoints.

**Why separate issuer definitions?**
- Centralizes JWKS endpoint configuration (avoid repetition)
- Makes it easy to support multiple OIDC providers (GitHub, GitLab, custom)
- Allows caching configuration per issuer
- Easy to update if provider changes endpoints

**Fields:**
- `issuer_url`: The expected `iss` claim value in JWT tokens
- `jwks_url`: Where to fetch public keys for JWT signature verification

---

### Server Configuration

**Purpose:** Define Galaxy/Automation Hub server endpoints and authentication mechanisms.

**Fields:**
- `base_url`: Base URL for the Galaxy server (everything before `/api/v3/...`)
- `token`: Environment variable name containing the Galaxy API token (for token-based auth)
- `oauth_secret`: OAuth configuration (for OAuth-based auth like Automation Hub)
  - `auth_url`: Token endpoint for OAuth client credentials flow
  - `client_id`: OAuth client ID
  - `client_secret`: Environment variable name containing the OAuth client secret

_Either `token` or `oauth_secret` is required (not both)._

**Token Selection Logic:**

When proxy needs to make authenticated request:
1. Determine required server from URL (`server_id`)
2. Look up `servers[server_id]` in configuration
3. If server has `token`:
   - Load token value from environment variable
   - Use for `Authorization: Token <value>` header
4. If server has `oauth_secret`:
   - Load `client_id` and `client_secret` from environment
   - Exchange credentials at `auth_url` for access token (no caching, exchange on every request)
   - Use access token for `Authorization: Bearer <access_token>` header
5. Proxy request to Galaxy/Automation Hub with authentication

**Note:** Tokens are NOT scoped to namespaces. Each server has a single token with access to all namespaces needed. Namespace authorization is enforced through `allowed_collections` in authorization rules during publish.

---

### Client Authorization Configuration

**Purpose:** Define the client authorization rules and what servers/collections it can publish to.

**Fields:**
- `oidc_issuer`: OIDC issuer identifier (references key in `oidc_issuers` section)
- `claims`: JWT claims that must match (supports exact match and wildcards with `*`)
  - Claim names depend on the issuer and what is present in the token
  - Wildcard matching: `repository: myorg/*` matches any repo in myorg
  - Tag patterns: `ref: refs/tags/v*` matches tags starting with 'v'
- `servers`: Map of server IDs to allowed collections
  - Key: server ID (from `servers` section)
  - Value: `allowed_collections` list

**Claim Matching:**

Claims support exact match and wildcard (`*`) patterns:

```yaml
# Exact match
claims:
  repository: myorg/specific-repo

# Wildcard suffix
claims:
  repository: myorg/*  # Matches myorg/repo1, myorg/repo2, etc.

# Wildcard in ref
claims:
  ref: refs/tags/*  # Any tag
  ref: refs/tags/v*  # Tags starting with 'v'

# Multiple claims (all must match)
claims:
  repository: myorg/repo
  ref: refs/heads/main
  workflow: .github/workflows/publish.yml
```

**Authorization Checks (in order):**

**Phase 1: JWT Validation**

1. **Validate JWT per the doc above**

**Phase 2: Authorization Rules**

2. **Find Matching Authorization Rule:**
   - Iterate through `authorization[]` rules
   - For each rule:
     - Check: Does `rule.oidc_issuer` match the issuer identifier found in step 3?
     - Check: Do ALL claims in `rule.claims` match JWT claims?
       - Support exact match: `repository: myorg/repo`
       - Support wildcards: `repository: myorg/*`, `ref: refs/tags/v*`
   - Use first matching rule
   - If no match: **403 Forbidden** (no authorization rule for this token/claims)

3. **Verify Server Access:**
   - Check: Is requested `server_id` in `rule.servers` map?
   - If not: **403 Forbidden** (not authorized for this server)

**Phase 3: Collection Validation (PUBLISH ONLY)**

4. **Check Endpoint Type:**
   - If request is `POST /api/v3/artifacts/collections/` → Publish endpoint, continue to step 5
   - If request is `GET /api/` → API discovery endpoint, skip to Phase 4 (no collection validation)
   - If request is `GET /api/v3/imports/collections/{task_id}/` → Import status endpoint, skip to Phase 4 (no collection validation)

5. **Extract Collection Metadata (Publish Only):**
   - Buffer/stream request body (tarball)
   - Extract `MANIFEST.json` from tarball
   - Parse namespace and name from `MANIFEST.json`
   - Format: `{"collection_info": {"namespace": "myorg", "name": "mycollection"}}`

6. **Validate Collection Name (Publish Only):**
   - Get `allowed_collections` from authorization rule
   - Check if `{namespace}.{name}` exactly matches any entry in `allowed_collections`
   - **Exact match only**: `"myorg.mycollection"` (wildcards NOT supported)
   - If no match: **403 Forbidden** (collection not allowed for this repository)
   - **This prevents compromised repos from publishing unauthorized collections**

**Phase 4: Token Selection**

7. **Get Galaxy/Hub Token:**
   - Get server config: `servers[server_id]`
   - If server has `token`:
     - Load token value from environment variable
     - Use for `Authorization: Token <value>` header
   - If server has `oauth_secret`:
     - Load `client_id` and `client_secret` from environment
     - Exchange at `auth_url` for access token (no caching)
     - Use for `Authorization: Bearer <access_token>` header
   - If server not found or missing auth config: **500 Internal Server Error** (misconfiguration)

**Phase 5: Proceed to Proxying**

8. **Success:**
   - ✅ JWT validated and signature verified
   - ✅ Authorization rule matched
   - ✅ Server access confirmed
   - ✅ Collection name authorized (if publish endpoint only)
   - ✅ Token retrieved for Galaxy authentication
   - **Proceed to proxy request to Galaxy/Automation Hub**

**Summary:**
- `GET /api/` - Steps 1-3, 7-8 (API discovery, no collection validation)
- `POST /api/v3/artifacts/collections/` - Steps 1-8 (full validation including collection name)
- `GET /api/v3/imports/collections/{task_id}/` - Steps 1-3, 7-8 (import status check, no collection validation)

---

**Collection Validation (Why It's Important):**

**Without collection name validation:**
- Any authorized repo could publish to any collection the proxy has tokens for
- Example: `myorg/repo-a` could publish `myorg.malicious` collection

**With collection name validation:**
- `ansible-collections/community.general` can ONLY publish `community.general` collection
- Collection names must be explicitly listed in `allowed_collections` (exact matches only)
- Collection name extracted from upload's `MANIFEST.json` and validated before proxying
- **No wildcard support**: Each collection must be explicitly listed (e.g., `myorg.collection1`, `myorg.collection2`)

## Client Usage

### Using ansible-galaxy CLI

**Publish Collection:**
```bash
# Build collection (done locally)
ansible-galaxy collection build

# Get OIDC token from GitHub Actions
AUDIENCE=https://our-proxy.example.com
OIDC_TOKEN=$(curl -H "Authorization: bearer ${ACTIONS_ID_TOKEN_REQUEST_TOKEN}" \
    "${ACTIONS_ID_TOKEN_REQUEST_URL}&audience=${AUDIENCE}" | jq -r '.value')

# Publish using our proxy
ansible-galaxy collection publish \
  namespace-collection-1.0.0.tar.gz \
  --server https://our-proxy.example.com/api/v1/galaxy/ \
  --token $OIDC_TOKEN
```

**Notes:**
- CLI automatically appends `api/v3/artifacts/collections/` to server URL
- So full URL becomes: `https://our-proxy.example.com/api/v1/galaxy/api/v3/artifacts/collections/`
- Our proxy extracts `galaxy` as server_id and `/api/v3/artifacts/collections/` as galaxy_path

### Publishing to Different Servers

**Ansible Galaxy (community):**
```bash
--server https://our-proxy.example.com/api/v1/galaxy/
```

**Red Hat Automation Hub:**
```bash
--server https://our-proxy.example.com/api/v1/automation-hub/
```

### GitHub Actions Workflow

**Workflow File:** `.github/workflows/publish.yml`

```yaml
name: Publish Collection

on:
  release:
    types: [published]

permissions:
  id-token: write  # Required for OIDC token
  contents: read

jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build Collection
        run: ansible-galaxy collection build

      # This could be condensed down to a single task that does both the OIDC get
      # and ansible-galaxy collection publish ...
      - name: Get OIDC Token
        id: oidc
        run: |
          OIDC_TOKEN=$(curl -H "Authorization: bearer $ACTIONS_ID_TOKEN_REQUEST_TOKEN" \
            "$ACTIONS_ID_TOKEN_REQUEST_URL&audience=https://our-proxy.example.com" \
            | jq -r '.value')
          echo "::add-mask::$OIDC_TOKEN"
          echo "token=$OIDC_TOKEN" >> $GITHUB_OUTPUT

      - name: Publish to Galaxy
        run: |
          ansible-galaxy collection publish *.tar.gz \
            --server https://our-proxy.example.com/api/v1/galaxy/ \
            --token ${{ steps.oidc.outputs.token }}
```

**What Happens:**
1. Workflow builds collection locally
2. Requests OIDC token from GitHub (audience = our proxy URL)
3. Calls `ansible-galaxy collection publish` with our proxy URL and OIDC token
4. `ansible-galaxy` makes POST request to our proxy
5. Proxy validates OIDC, proxies to Galaxy, returns result
6. `ansible-galaxy` polls task status through our proxy until complete

## Deployment

### Platform: AWS App Runner

**AWS App Runner** is chosen as the deployment platform for this proxy service.

**Why App Runner:**
- ✅ **Serverless compute** - No EC2 instances to manage or patch
- ✅ **No payload size limits** - Supports large collection tarballs (10MB+)
- ✅ **Simple deployment** - Deploy directly from source code (no Dockerfile required)
- ✅ **Auto-scaling** - Automatically scales instances based on request load
- ✅ **Built-in HTTPS** - Automatic TLS certificates and load balancing
- ✅ **Cost-effective** - ~$6-10/month for low-volume use cases
- ✅ **CloudWatch integration** - Automatic logging and metrics

**vs Lambda:**
- Lambda has 10MB API Gateway payload limit (too small for larger collections)
- App Runner supports much larger payloads with no special configuration

**vs ECS Fargate:**
- Fargate requires ALB configuration (~$16/month fixed cost)
- App Runner includes load balancing automatically
- App Runner is 4-5x cheaper for low-traffic workloads

### Architecture Overview

```
GitHub Actions Workflow
        ↓
    HTTPS Request (with OIDC token)
        ↓
AWS App Runner Service
    ├── Auto-scaling instances (1-N containers)
    ├── Built-in HTTPS load balancer
    ├── CloudWatch Logs
    └── Environment variables → AWS Secrets Manager
        ↓
    Proxy Logic (Python FastAPI)
        ├── Validate OIDC token (JWKS cached in-memory)
        ├── Check authorization rules
        ├── Validate collection (for publish)
        ├── Get Galaxy/Hub token from Secrets Manager
        ├── Exchange OAuth credentials for Hub token (cached in-memory)
        └── Proxy to Galaxy/Automation Hub
            ↓
Galaxy / Automation Hub Server
```

### Secrets Management

**AWS Secrets Manager** is used for storing sensitive Galaxy/Automation Hub tokens.

**Benefits:**
- ✅ Encrypted at rest and in transit
- ✅ Fine-grained IAM access control
- ✅ Audit trail (CloudTrail logs all access)
- ✅ Rotation support (rotate tokens without redeploying)
- ✅ Native App Runner integration

**Configuration:**

```bash
# Create secrets
aws secretsmanager create-secret \
  --name galaxy-publisher/galaxy-token \
  --description "Ansible Galaxy API token" \
  --secret-string "your-galaxy-api-token-here"

aws secretsmanager create-secret \
  --name galaxy-publisher/automation-hub-secret \
  --description "Red Hat Automation Hub OAuth client secret" \
  --secret-string "your-oauth-client-secret-here"
```

**App Runner Configuration:**

Environment variables reference Secrets Manager ARNs:
```json
{
  "GALAXY_TOKEN": "arn:aws:secretsmanager:us-east-1:123456789:secret:galaxy-publisher/galaxy-token",
  "AUTOMATION_HUB_SECRET": "arn:aws:secretsmanager:us-east-1:123456789:secret:galaxy-publisher/automation-hub-secret"
}
```

App Runner automatically:
1. Resolves the ARN
2. Fetches the secret value from Secrets Manager
3. Provides the actual secret value to the application as `os.environ['GALAXY_TOKEN']`

**Python Usage:**
```python
import os

# No boto3 needed - App Runner handles resolution
galaxy_token = os.environ['GALAXY_TOKEN']  # Actual token, not ARN
automation_hub_secret = os.environ['AUTOMATION_HUB_SECRET']
```

**Cost:** $0.40/secret/month = $0.80/month for 2 secrets

### Caching Strategy

**In-Memory Caching** is used for performance optimization without external dependencies.

#### JWKS Caching (GitHub Public Keys)

**Purpose:** Avoid fetching GitHub's public keys on every request

**Implementation:**
```python
class JWKSCache:
    def __init__(self):
        self._cache = {}

    def get(self, issuer_url: str, jwks_url: str, ttl: int = 86400) -> dict:
        # Check cache (24 hour TTL)
        if issuer_url in self._cache:
            jwks, timestamp = self._cache[issuer_url]
            if time.time() - timestamp < ttl:
                return jwks  # Cache hit

        # Cache miss - fetch from GitHub
        response = httpx.get(jwks_url)
        jwks = response.json()
        self._cache[issuer_url] = (jwks, time.time())
        return jwks

# Global instance (persists across requests in same container)
jwks_cache = JWKSCache()
```

**Characteristics:**
- ✅ Per-instance cache (not shared across containers)
- ✅ 24-hour TTL (JWKS rarely changes)
- ✅ Cache miss = ~100-200ms penalty to fetch from GitHub
- ✅ Free (no external service)

**Cache Hit Rate:** 90%+ (JWKS fetched once per day per instance)

#### OAuth Token Caching (Automation Hub)

**Purpose:** Avoid exchanging OAuth credentials on every request

**Implementation:**
```python
class OAuthTokenCache:
    def __init__(self):
        self._tokens = {}

    def get_token(self, server_id: str, client_id: str,
                  client_secret: str, auth_url: str) -> str:
        cache_key = f"{server_id}:{client_id}"

        # Check cache (with 5 min expiry buffer)
        if cache_key in self._tokens:
            token, expires_at = self._tokens[cache_key]
            if time.time() < expires_at - 300:
                return token  # Cache hit

        # Cache miss - exchange credentials
        response = httpx.post(auth_url, data={
            'grant_type': 'client_credentials',
            'client_id': client_id,
            'client_secret': client_secret
        })
        token_data = response.json()

        # Cache with expiry (typically 1 hour)
        access_token = token_data['access_token']
        expires_in = token_data.get('expires_in', 3600)
        expires_at = time.time() + expires_in
        self._tokens[cache_key] = (access_token, expires_at)

        return access_token

oauth_cache = OAuthTokenCache()
```

**Characteristics:**
- ✅ Per-instance cache (not shared across containers)
- ✅ 1-hour token lifetime (typical OAuth access token TTL)
- ✅ Cache miss = ~200-500ms penalty for OAuth exchange
- ✅ Free (no external service)

**Cache Hit Rate:** 80-90% for typical usage (2-5 publishes/day)

#### Why Not DynamoDB or ElastiCache?

**For MVP, in-memory caching is sufficient because:**
- Low request volume (2-5 publishes per day/week)
- Cache miss penalty is acceptable (~200-500ms)
- Simplifies architecture (no additional AWS services)
- Cost savings ($0 vs $15-30/month for ElastiCache or DynamoDB)

**Future optimization:** If volume increases (>100 publishes/day) or multiple App Runner instances are needed, consider DynamoDB for shared caching.

**Configuration Models:**
```python
# src/galaxy_publisher/models.py
from dataclasses import dataclass
from typing import Optional

@dataclass
class OIDCIssuer:
    issuer_url: str
    jwks_url: str

@dataclass
class ServerTokenAuth:
    token: str  # Environment variable name

@dataclass
class OAuthSecret:
    auth_url: str
    client_id: str
    client_secret: str  # Environment variable name

@dataclass
class Server:
    base_url: str
    token: Optional[ServerTokenAuth] = None
    oauth_secret: Optional[OAuthSecret] = None

    def __post_init__(self):
        if not self.token and not self.oauth_secret:
            raise ValueError("Server must have either token or oauth_secret")
        if self.token and self.oauth_secret:
            raise ValueError("Server cannot have both token and oauth_secret")

@dataclass
class AuthorizationRule:
    oidc_issuer: str
    claims: dict[str, str]
    servers: dict[str, dict[str, list[str]]]  # server_id -> {allowed_collections: [...]}

@dataclass
class Settings:
    oidc_audience: str

@dataclass
class Config:
    version: int
    settings: Settings
    oidc_issuers: dict[str, OIDCIssuer]
    servers: dict[str, Server]
    authorization: list[AuthorizationRule]

    def __post_init__(self):
        if self.version != 1:
            raise ValueError(f"Unsupported config version: {self.version}")
```

**Configuration Loading (with validation):**
```python
# src/galaxy_publisher/config.py
import os
from pathlib import Path
from typing import Any

import yaml

from galaxy_publisher.models import (
    AuthorizationRule,
    Config,
    OAuthSecret,
    OIDCIssuer,
    Server,
    ServerTokenAuth,
    Settings,
)


def load_config(config_path: str | Path) -> Config:
    """Load and validate configuration from YAML file.

    Note: YAML is a superset of JSON, so JSON config files also work.
    """
    with open(config_path) as f:
        data = yaml.safe_load(f)

    return _dict_to_config(data)


def _dict_to_config(data: dict[str, Any]) -> Config:
    """Convert dictionary to Config dataclass with validation."""
    # Parse settings
    settings_data = data.get("settings", {})
    settings = Settings(
        oidc_audience=settings_data["oidc_audience"]
    )

    # Parse OIDC issuers
    oidc_issuers = {}
    for issuer_id, issuer_data in data.get("oidc_issuers", {}).items():
        oidc_issuers[issuer_id] = OIDCIssuer(
            issuer_url=issuer_data["issuer_url"],
            jwks_url=issuer_data["jwks_url"]
        )

    # Parse servers
    servers = {}
    for server_id, server_data in data.get("servers", {}).items():
        token_auth = None
        oauth_secret = None

        if "token" in server_data:
            token_auth = ServerTokenAuth(token=server_data["token"])

        if "oauth_secret" in server_data:
            oauth_data = server_data["oauth_secret"]
            oauth_secret = OAuthSecret(
                auth_url=oauth_data["auth_url"],
                client_id=oauth_data["client_id"],
                client_secret=oauth_data["client_secret"]
            )

        servers[server_id] = Server(
            base_url=server_data["base_url"],
            token=token_auth,
            oauth_secret=oauth_secret
        )

    # Parse authorization rules
    authorization = []
    for rule_data in data.get("authorization", []):
        authorization.append(AuthorizationRule(
            oidc_issuer=rule_data["oidc_issuer"],
            claims=rule_data["claims"],
            servers=rule_data["servers"]
        ))

    # Create and validate config
    config = Config(
        version=data["version"],
        settings=settings,
        oidc_issuers=oidc_issuers,
        servers=servers,
        authorization=authorization
    )

    return config


# Load configuration at startup
config_path = os.getenv("CONFIG_PATH", "config/servers.yml")
config = load_config(config_path)
```

App Runner automatically scales based on:
- **Concurrent requests** (default: 100 requests per instance)
- **CPU utilization**
- **Memory utilization**

- Max instances: 25

- Scale-down: When traffic decreases, after cooldown period

### Security Considerations

**IAM Permissions:**
- App Runner instance role needs `secretsmanager:GetSecretValue` for configured secrets
- Principle of least privilege: Only grant access to `galaxy-publisher/*` secrets

**Network Security:**
- App Runner services are public by default (HTTPS only)
- Consider VPC connector if accessing private resources (not needed for this use case)
- TLS 1.2+ enforced automatically

**Secret Rotation:**
```bash
# Rotate Galaxy token
aws secretsmanager update-secret \
  --secret-id galaxy-publisher/galaxy-token \
  --secret-string "new-token-here"

# Trigger App Runner deployment to pick up new secret
aws apprunner start-deployment --service-arn <arn>
```

**Audit Trail:**
- All secret accesses logged to CloudTrail
- All API requests logged to CloudWatch Logs
- App Runner deployments logged to CloudWatch Events

### Troubleshooting

**Common Issues:**

**1. Service fails to start:**
```bash
# Check build logs
aws apprunner list-operations --service-arn <arn>
aws logs tail /aws/apprunner/<service-name>/<service-id>/system --follow
```

**2. Secrets not loading:**
- Verify IAM role has `secretsmanager:GetSecretValue` permission
- Check secret ARN is correct in environment variables
- Verify secret exists in same region as App Runner service

**3. OIDC validation failing:**
- Check JWKS cache (may need to clear on OIDC provider key rotation)
- Verify `iss` claim matches configured issuer URL
- Check audience claim matches `OIDC_AUDIENCE` environment variable

**4. High latency:**
- Check if OAuth token caching is working (should see cache hit logs)
- Monitor `ActiveInstances` metric (may need more instances)
- Check App Runner metrics for CPU/memory bottlenecks

### JWT Token Generation for Tests

**Purpose:** Generate valid OIDC tokens signed with a test private key, and provide a JWKS endpoint for the proxy to validate them.

**Implementation:** `tests/jwt_utils.py`

```python
"""JWT token generation and validation utilities for testing."""
import json
import time
from pathlib import Path

from cryptography.hazmat.primitives import serialization
from cryptography.hazmat.primitives.asymmetric import rsa
import jwt


class JWTTestHelper:
    """Helper for generating test JWT tokens and JWKS."""

    def __init__(self, keys_dir: Path | None = None):
        """Initialize with RSA key pair."""
        if keys_dir is None:
            keys_dir = Path(__file__).parent / "fixtures" / "keys"

        keys_dir.mkdir(parents=True, exist_ok=True)

        self.private_key_path = keys_dir / "test_private_key.pem"
        self.public_key_path = keys_dir / "test_public_key.pem"

        # Generate or load keys
        if not self.private_key_path.exists():
            self._generate_keys()

        with open(self.private_key_path, "rb") as f:
            self.private_key = serialization.load_pem_private_key(
                f.read(), password=None
            )

        with open(self.public_key_path, "rb") as f:
            self.public_key = serialization.load_pem_public_key(
                f.read()
            )

    def _generate_keys(self) -> None:
        """Generate RSA key pair for signing test JWTs."""
        print(f"Generating test RSA keys at {self.private_key_path.parent}")

        # Generate private key
        private_key = rsa.generate_private_key(
            public_exponent=65537,
            key_size=2048,
        )

        # Save private key
        with open(self.private_key_path, "wb") as f:
            f.write(private_key.private_bytes(
                encoding=serialization.Encoding.PEM,
                format=serialization.PrivateFormat.PKCS8,
                encryption_algorithm=serialization.NoEncryption()
            ))

        # Save public key
        public_key = private_key.public_key()
        with open(self.public_key_path, "wb") as f:
            f.write(public_key.public_bytes(
                encoding=serialization.Encoding.PEM,
                format=serialization.PublicFormat.SubjectPublicKeyInfo
            ))

    def create_token(
        self,
        repository: str = "test-org/test-collection",
        ref: str = "refs/heads/main",
        aud: str = "http://localhost:8000",
        iss: str = "http://localhost:8002",
        exp_minutes: int = 60,
        **extra_claims
    ) -> str:
        """Create a signed JWT token for testing."""
        now = int(time.time())

        claims = {
            "iss": iss,
            "aud": aud,
            "sub": f"repo:{repository}:ref:{ref}",
            "repository": repository,
            "ref": ref,
            "exp": now + (exp_minutes * 60),
            "nbf": now,
            "iat": now,
            **extra_claims
        }

        token = jwt.encode(
            claims,
            self.private_key,
            algorithm="RS256",
            headers={"kid": "test-key-1"}
        )

        return token

    def create_expired_token(self, **kwargs) -> str:
        """Create an expired token for testing."""
        return self.create_token(exp_minutes=-10, **kwargs)

    def create_invalid_signature_token(self, **kwargs) -> str:
        """Create a token with invalid signature."""
        # Create a token with a different key
        bad_key = rsa.generate_private_key(
            public_exponent=65537,
            key_size=2048,
        )

        claims = {
            "iss": kwargs.get("iss", "http://localhost:8002"),
            "aud": kwargs.get("aud", "http://localhost:8000"),
            "repository": kwargs.get("repository", "test-org/test-collection"),
            "exp": int(time.time()) + 3600,
        }

        return jwt.encode(claims, bad_key, algorithm="RS256")

    def get_jwks(self) -> dict:
        """Generate JWKS (JSON Web Key Set) for token validation."""
        # Get public key in JWK format
        public_numbers = self.public_key.public_numbers()

        # Convert to base64url format
        def int_to_base64url(n: int) -> str:
            """Convert integer to base64url string."""
            import base64
            byte_length = (n.bit_length() + 7) // 8
            n_bytes = n.to_bytes(byte_length, byteorder='big')
            return base64.urlsafe_b64encode(n_bytes).decode('utf-8').rstrip('=')

        jwks = {
            "keys": [
                {
                    "kty": "RSA",
                    "use": "sig",
                    "kid": "test-key-1",
                    "alg": "RS256",
                    "n": int_to_base64url(public_numbers.n),
                    "e": int_to_base64url(public_numbers.e)
                }
            ]
        }

        return jwks
```

### CI/CD Integration

**GitHub Actions Workflow:** `.github/workflows/test.yml`

```yaml
name: Test

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Install uv
        uses: astral-sh/setup-uv@v4
        with:
          version: "latest"

      - name: Set up Python
        run: uv python install 3.12

      - name: Install dependencies
        run: uv sync

      - name: Install ansible-galaxy CLI
        run: |
          sudo apt-get update
          sudo apt-get install -y ansible

      - name: Run tests
        run: uv run pytest -v --cov=galaxy_publisher --cov-report=xml

      - name: Upload coverage to Codecov
        uses: codecov/codecov-action@v4
        with:
          file: ./coverage.xml
          fail_ci_if_error: false

  lint:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Install uv
        uses: astral-sh/setup-uv@v4

      - name: Set up Python
        run: uv python install 3.12

      - name: Install dependencies
        run: uv sync

      - name: Check formatting with ruff
        run: uv run ruff format --check src/ tests/

      - name: Lint with ruff
        run: uv run ruff check src/ tests/

      - name: Type check with mypy
        run: uv run mypy src/
```
